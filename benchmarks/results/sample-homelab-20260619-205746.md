## Result

The run is meaningful but partial.

What was recorded:

- `staging` reached the homelab K3s control plane at `192.168.50.132:6443`.
- Nodes `homelab-01` and `homelab-02` were Ready.
- The namespace-scoped GHCR pull secret allowed the homelab to pull
  `ghcr.io/guaracloud/e-navigator:sha-8ab271c`.
- The Helm release rendered the DaemonSet with `imagePullSecrets:
  ghcr-pull-secret` and `privileged: false`.
- The DaemonSet rolled out successfully with one Ready E-Navigator pod on each
  homelab node:
  - `e-navigator-bench-wbncb` on `homelab-01`
  - `e-navigator-bench-cmxkb` on `homelab-02`
- Both E-Navigator pods used digest
  `sha256:249ad67fa8578ade9ecc1279bcf52a52ae6038a342b7c68844ebfd7a38d4e34e`
  and had zero restarts in the post-run snapshot.
- The controlled workload completed on `homelab-01` with 60 loop iterations.
  Each loop emitted an exec marker and attempted DNS and HTTP calls against
  Kubernetes service endpoints.
- `kubectl top pods --containers` returned E-Navigator resource samples:
  - `e-navigator-bench-cmxkb`: `12m` CPU, `36Mi` memory
  - `e-navigator-bench-wbncb`: `36m` CPU, `46Mi` memory
- E-Navigator logs included:
  - `1,390` `source.aya_exec` records
  - `1,498` `source.aya_network` records
  - `532` `source.host_resource` records
  - derived network metrics, dependency edges, trace service paths, runtime
    security findings, and host resource metrics

What was inconclusive or absent:

- The E-Navigator logs did not include workload-attributed records for
  `e-navigator-bench-workload`.
- The workload generated DNS and HTTP opportunities, but no DNS signal records
  were present in `logs-post.txt`.
- No `source.aya_cpu_profile` records were present.
- Host resource logs included repeated warnings with `warning_count=2`; the
  warning details were not expanded by the current log output.
- Events still contain older `ImagePullBackOff` warnings for the previous
  `main` image attempt in the same namespace. The post-run DaemonSet pods for
  `sha-8ab271c` were Ready with zero restarts.

## Proof Boundary

This run proves that, in the `staging` homelab context, E-Navigator can be
pulled from GHCR with a namespace-scoped pull secret, deployed as the prepared
DaemonSet in `e-navigator-bench`, become Ready on both homelab nodes, emit Aya
exec and Aya network records, emit host resource records from mounted host
paths, run alongside a controlled workload, and produce a point-in-time
container CPU/RSS sample.

This run does not prove:

- DNS packet capture;
- CPU profiling runtime behavior for this run;
- workload-attributed capture for the controlled benchmark pod;
- OTLP, Prometheus, Pyroscope, Tempo, or production export behavior;
- Beyla, Alloy, Tempo, Prometheus, or Pyroscope replacement readiness;
- sustained overhead baselines or reduced overhead;
- reduced-privilege eBPF hardening beyond the recorded chart security context;
- production readiness.
