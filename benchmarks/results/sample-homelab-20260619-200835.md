# Homelab Validation Sample: 20260619-200835

This is a curated summary of the raw artifacts in
`benchmarks/results/20260619-200835/`.

## Scope

- Context: `staging`
- Cluster observed: `homelab-01` and `homelab-02`
- Namespace: `e-navigator-bench`
- Release: `e-navigator-bench`
- Image requested: `ghcr.io/guaracloud/e-navigator:main`
- Workload image: `docker.io/busybox:1.36`

The run used the guarded apply-and-collect path:

```bash
E_NAVIGATOR_HOMELAB_CONFIRM=1 \
E_NAVIGATOR_HOMELAB_APPLY=1 \
E_NAVIGATOR_HOMELAB_CONTEXT=staging \

E_NAVIGATOR_HOMELAB_IMAGE_REPOSITORY=ghcr.io/guaracloud/e-navigator \
E_NAVIGATOR_HOMELAB_IMAGE_TAG=main \
benchmarks/runner/homelab-collect.sh
```

## Result

The run is partial and does not prove E-Navigator runtime behavior.

What was recorded:

- `staging` reached the homelab K3s control plane at `192.168.50.132:6443`.
- Nodes `homelab-01` and `homelab-02` were Ready.
- The `e-navigator-bench` namespace was created.
- The Helm release installed a DaemonSet targeting
  `ghcr.io/guaracloud/e-navigator:main`.
- DaemonSet pods scheduled onto both homelab nodes:
  - `e-navigator-bench-ch94p` on `homelab-01`
  - `e-navigator-bench-pzxmz` on `homelab-02`
- The controlled workload completed successfully on `homelab-01` with 60 loop
  iterations. Each loop emitted an exec marker and attempted DNS and HTTP
  calls against Kubernetes service endpoints.
- The E-Navigator pods had zero restarts, but only because their containers
  never started.

What failed:

- Both E-Navigator pods stayed in `ImagePullBackOff`.
- The DaemonSet rollout timed out with `0 of 2 updated pods are available`.
- Homelab kubelet pulls for `ghcr.io/guaracloud/e-navigator:main` failed with
  GHCR authorization `403 Forbidden`.
- `kubectl logs` for E-Navigator returned only the waiting-container error.
- `kubectl top pods --containers` could not report E-Navigator CPU or memory
  because the E-Navigator containers were not running.

## Proof Boundary

This run proves only that the benchmark harness can install the namespace,
render the Helm release, schedule the DaemonSet pods, run the controlled
workload, and capture raw homelab evidence in the result directory.

This run does not prove:

- E-Navigator container startup;
- DaemonSet readiness;
- Aya exec or network event observation;
- DNS packet capture;
- host resource source runtime behavior;
- CPU/RSS overhead for E-Navigator;
- OTLP, Prometheus, Pyroscope, Tempo, or production export behavior;
- Beyla, Alloy, Tempo, Prometheus, or Pyroscope replacement readiness;
- reduced-privilege eBPF operation.

The next live attempt needs a homelab-pullable E-Navigator image or an approved
registry credential/image publication step before rerunning the guarded
collector.
