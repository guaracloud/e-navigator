Runtime shape

01

Sources observe

02

Generators summarize

Kubernetes path

Start with manifest proof, then deploy on a capable Linux cluster.

Validate manifests

Run client-side Kubernetes validation before applying privileged DaemonSet resources.

kubectl apply --dry-run=client -f deploy/kubernetes/namespace.yaml
kubectl apply --dry-run=client -f deploy/kubernetes/rbac.yaml
kubectl apply --dry-run=client -f deploy/kubernetes/configmap.yaml
kubectl apply --dry-run=client -f deploy/kubernetes/daemonset.yaml

Apply deliberately

Use a real Linux node with the documented eBPF, perf-event, tracefs, and container attribution prerequisites.

kubectl apply -f deploy/kubernetes/namespace.yaml
kubectl apply -f deploy/kubernetes/rbac.yaml
kubectl apply -f deploy/kubernetes/configmap.yaml
kubectl apply -f deploy/kubernetes/daemonset.yaml

Synthetic local run

Exercise the pipeline without privileged Linux or Kubernetes dependencies.

cargo run --locked -p e-navigator-cli -- --source synthetic

Proof boundaries

Evidence buckets stay separate.

Synthetic fixtures, Docker smoke, Kubernetes dry-runs, and privileged eBPF runtime proof are not interchangeable. The claims matrix is the source of truth for what is implemented and proven.

Vision

Consolidate the signal stack without hiding the gaps.

  1. Observe runtime behavior with bounded Rust and eBPF sources.
  2. Attribute signals to host, process, container, and Kubernetes context.
  3. Generate low-cardinality metrics, dependency edges, request spans, and profile windows.
  4. Keep OpenTelemetry export, storage, UI, flamegraphs, and live DNS capture explicit future work.
