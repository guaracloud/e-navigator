# Product

## Register

brand

## Users

E-Navigator serves platform engineers, site reliability engineers, security
engineers, and Rust contributors who evaluate, deploy, operate, or extend a
node-local observability agent. They need to understand what the agent captures,
what it costs, how it fails, and which claims have runtime evidence before they
place it on Linux or Kubernetes nodes.

## Product Purpose

E-Navigator turns bounded Linux and Kubernetes runtime observations into native,
versioned metrics, traces, profiles, topology, and security signals. Its public
surface must help an operator move from evaluation to a verified, scoped,
low-overhead deployment without confusing implemented behavior with proven
runtime behavior. Success means that a reader can choose the right installation,
configure only the signals they need, verify health and loss accounting, and
find the exact evidence or boundary behind every material claim.

## Brand Personality

Rigorous, candid, and engineered. The voice is confident because it shows
contracts, commands, limits, and evidence. It avoids hype, vague superlatives,
and unexplained jargon.

## Anti-references

- Generic SaaS landing pages that replace technical substance with slogans,
  decorative metric cards, or unsupported performance claims.
- Cyberpunk and terminal-cosplay interfaces that use visual noise as shorthand
  for systems engineering.
- Documentation portals that hide the runnable path behind deep navigation or
  repeat the same content without a clear source of truth.
- Vendor-compatibility language that weakens E-Navigator's native contracts or
  implies replacement of Pyroscope as the profile backend.

## Design Principles

1. Put proof before promise, and link every consequential claim to its evidence
   level or explicit boundary.
2. Make the golden path visible from the first screen and usable without prior
   knowledge of the repository.
3. Use progressive disclosure, with concise decisions first and exact commands,
   limits, and failure behavior close behind.
4. Treat low overhead as an operational discipline based on scoped capture,
   bounded state, measurable loss, and representative benchmarks.
5. Keep the website, README, Markdown guides, CLI, chart, and runtime defaults in
   agreement.

## Accessibility & Inclusion

Target WCAG 2.2 AA. All content must remain usable with keyboard navigation,
visible focus, semantic landmarks, sufficient contrast, zoom, narrow viewports,
reduced motion, and horizontal scrolling for code blocks. Color cannot be the
only carrier of evidence level, warning state, or action priority.
